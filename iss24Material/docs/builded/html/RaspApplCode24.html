Costruiamo applicazioni

SonarAndLed¶

  • Sistema SonarAndLed organizzato in funzioni Python

  • Ciascuna funzione ha una precisa responsbilità

  • La funzione doJob() funge da coordinatore

  • La funzione applLogic definisce la logica applicativa

  • La funzione forward invia informazioni via MQTT

Inizializzazione

 
import RPi.GPIO as GPIO
import time
import sys
import paho.mqtt.client as paho

### CONFIGURATION FOR LED
GPIO.setmode(GPIO.BCM)
GPIO.setup(25,GPIO.OUT)

### CONFIGURATION FOR SONAR
GPIO.setmode(GPIO.BCM)
GPIO.setwarnings(False)
TRIG = 17
ECHO = 27
GPIO.setup(TRIG,GPIO.OUT)
GPIO.setup(ECHO,GPIO.IN)

### MQTT
brokerAddr="mqtt.eclipseprojects.io"
msg       = "msg(sonardata,event,sonar,none,distance(D),N)"
n         = 1
client    = paho.Client(paho.CallbackAPIVersion.VERSION1,"sonarAndLed")


def init():
    GPIO.output(TRIG, False)   #TRIG parte LOW
    client.connect(brokerAddr, 1883, 60)
    print ('Waiting a few seconds for the sensor to settle')
    time.sleep(2)

def doJob():
    init()

Funzioni operative

 
def ledOn():
  GPIO.output(25,GPIO.HIGH)
  ##forward()  #QUI???

def ledOff():
  GPIO.output(25,GPIO.LOW)

def sonarWork():
  GPIO.output(TRIG, True)    #invia impulsoTRIG
  time.sleep(0.00001)
  GPIO.output(TRIG, False)

  #attendi che ECHO parta e memorizza tempo
  while GPIO.input(ECHO)==0:
      pulse_start = time.time()

  # register the last timestamp
  while GPIO.input(ECHO)==1:
      pulse_end = time.time()

  pulse_duration = pulse_end - pulse_start
  dist = pulse_duration * 17165   #distance = vt/2
  return dist

def forward(distance):
    global n
    n = n + 1
    client.publish("unibo/sonar/events",
     msg.replace("D",str(distance)).replace("N", str(n)))

Sonar e Led come attori¶

Il software di basso livello viene inglobato all’interno di attori qak che possono sfruttare la Qak infrastructure per interagire tra loro, in locale e/o in remoto, in modo technology-independent.

  1. Definiamo un sistena che permette l’accensione/spegimento remoto di un Led: ledalone.qak.

  2. Definiamo un sistena ad attori (Progetto unibo.sonarqak24) che utilizza il SONAR HC-SR04 per emettere informazioni di distanza filtrate e anche informazioni di livello applicativo, quali la rilevazione di un ostacolo.

Progetto unibo.sonarqak24¶

Sviluppiamo un sistema software che utilizza il SONAR HC-SR04 operando in diverse fasi:

  1. sonarqak24 Fase1: predisposizione della gestione dei dispositivi.

sonarqak24 Fase1¶

GOAL: Costruire un (sotto)sistema ad attori che utilizza il SONAR HC-SR04.

  1. Definizione del CodedQActor di nome sonar che opera come un simulatore del sonar (classe Kotlin sonarSimulator) o che esegue codice Python per la gestione di un sonar fisico SONAR HC-SR04 (classe Kotlin sonarHCSR04Support23.kt). Questo attore emmete l’evento locale sonar:distance(D) usando la primitiva emitLocalstream.

  2. Definizione dei CodedQActor di nome dataCleaner e distancefilter ciascuno dei quali opera come uno StreamedQActor. Pertanto, essi sono usabili per costruire una pipe che ha sonar come sorgente-dati e che provvede a eliminare dati spuri (dataCleaner) e a generare (distancefilter) eventi significativi per il livello applicativo.
